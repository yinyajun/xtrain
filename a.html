Cycle 06 — Overview

从 Ghostwriter 到 Signals

Cycle 04-05 的核心假设是 Agent 帮你写帖子、帮你发帖子。经过两个 cycle 的验证，我们发现这个方向有根本性问题：Agent 不知道用户今天经历了什么，生成的内容永远是围绕 VOICE.md 的变体，缺乏真实性。

Cycle 06 的方向转变：Agent 不帮你写，Agent 帮你看。 从 ghostwriter 变成 signal scout。Agent 持续扫描 X，把值得你注意的信号筛出来，你决定是否行动。

命名体系

产品里的每个概念需要一个清晰的名字。以下是决策：

页面名称
Signals — 页面展示的就是一串 signal，所以页面名和内容名一致。Sidebar 里显示为 「Signals」 + badge 数字。
卡片名称
Signal（信号）— Agent 检测到的每一个值得注意的信号就是一张卡片。「你有 5 个新 Signal」比「你有 5 个新 Insight」更有紧迫感。Insight 暗示深度思考，但大部分卡片是时效性的行动提示。
Signal 类型
每个 Signal 标记为四种类型之一：Engage（对话机会）、Create（内容时机）、Research（情报摘要）、Network（关系动态）。用颜色 badge 区分。
行动
每张 Signal 卡片的最终动作都是 导向 X：回复、发帖、查看主页、阅读。不在 Tomo 内完成，Tomo 是决策层，X 是执行层。
扫描
Scan — Agent 每次 heartbeat（~30min）执行一次 scan，扫描你的帖子回复 + 领域趋势 + 提及 + 互动数据。Signals 页顶部显示「12 分钟前扫描」。

命名决策记录

早期方案叫过 Radar（扫描器 metaphor），后来决定统一用 Signals。原因：软件产品一般用页面显示的内容命名（Inbox → Email 是例外因为 Email 被邮件占用）。我们的 signal 没有命名冲突，所以 页面和卡片用同一个词，认知负担最低。

为什么不叫别的

  • Radar — 更有记忆点，但多了一层 metaphor 抽象，用户需要学「Radar 里放 Signal」
  • Feed — 太 generic，和 X 自己的 feed 容易混淆
  • Pulse — 暗示心跳/健康，但我们检测的是外部信号，不是内部状态
  • Insights — 暗示深度分析，但 Engage 卡片是时效性的行动提示
  • Opportunities — 太 sales-y，而且 Research/Network 不是「机会」

四层信号模型

Layer 1 — Engage

参与对话

你自己帖子的回复（Inbound）+ 外部值得加入的对话（Outbound）。最高优先级，有时效性。

Layer 2 — Create

内容时机

趋势 × 你的角度 × idea backlog 的交叉匹配。告诉你「现在发什么效果最好」。

Layer 3 — Research CYCLE 6 暂缓

持续情报

用户设定关注主题，Agent 持续监控，定期输出 briefing。Dataminr 面向个人用户的版本。

Layer 4 — Network

关系信号

谁在持续关注你？谁最近互动减少了？从互动数据里提取关系动态。

Cycle 6 范围（4/20 sync）： Cycle 6 聚焦 Engage（Inbound + Outbound） 和 Create 两层。Research 暂缓，不进本 cycle 交付范围（UI 占位可保留灰化状态，后端不实现）。Network 作为低频信号仍在范围内。

Badge Taxonomy（Signal 卡片标签规范）

Signal 卡片上的每一个标签都要遵守固定规范。全部英文、固定 slot、颜色语义明确。实现时以这一节为准。

Slot 1 — Type Badge（必有，1 个，弱化的彩色 dot + 全大写小字）

每张 Signal 卡片必须有且只有一个 Type Badge。视觉上是弱化的元数据：一个小色点 + 全大写小字（如 ● ENGAGE），不是填充 pill。Badge 的存在是给系统做 filter 分类，不是给用户读内容。颜色 = Signal 类型，只用一级名称（Engage / Create / Research / Network），不再附加 · 子类型。子类型的差异由 section header 和 why-box 承担。

Badge 文案视觉含义子类型差异由谁承担
Engage (teal dot)● ENGAGE需要回应的对话机会（Inbound 自己帖子 + Outbound 外部）Section header：INBOX · WORTH REPLYING vs DISCOVER · WORTH JOINING；teal vs peach 色点保留视觉差异
Create (amber dot)● CREATE发帖时机Section header：CREATE · BEST TO POST TODAY；why-box 解释触发源（trend / idea / chat seed）
Research (blue dot)● RESEARCH周期性情报摘要Section header + why-box
Network (purple dot)● NETWORK关系动态Section header + why-box
为什么把 badge 弱化：之前的填充 pill（Engage · Inbox）会和卡片标题争视觉位。但用户读卡片靠的是「标题 + 内容 + why-box」这套人话叙事，不是系统分类词。Badge 是给 filter tab 做锚点的，所以视觉上应该退到二线。

为什么去掉子类型：「Inbox / Discover / Trend / Idea / Activity」对从没用过 Tomo 的人没有意义，需要先学系统词汇才能读懂。这些子类型的真正用途是后端分类（不同的 dedup_key、不同的 scan 频率），不是前端展示。前端只需要让用户知道「这是个对话机会 / 这是个发帖时机」，section header 已经够用。

CSS class 命名（保留）

虽然 badge 文案不再带子类型，但 CSS class 仍按子类型命名，因为不同子类型可能有不同的色调微差异（如 Engage A teal vs Engage B peach）：

Slot 2 — Trigger Chip（0-2 个，中性灰）

在卡片扫一眼时立刻传达「为什么现在、为什么是你」的关键事实。原则：只保留不需要解释就能读懂的 chip。 凡是需要了解系统内部才能读懂的标签（数据来源、内部流转路径），一律移入 why-box 用完整句子说明，不出现在 chip 上。

Chip出现场景替换/移除说明
you were mentioned你被 @ 点名原 @mentioned → 改为完整短语，立即可读
quoted by @handle你的帖子被某人引用原 quoted → 带出具体用户名，更有意义
N new repliesEngage · Inbox 累计新回复数（示例：3 new replies）保留，已是用户语言
trending fast话题热度在快速上升原 velocity +N% → 百分比是系统指标，用户不关心具体数字，关心「这个话题正在爆」
~Nh windowCreate Signal 的时机窗口倒计时（示例：~6h window）保留，时效感清晰
移除的 chip（原因）：
  • high-value reply — 「高价值」是系统的判断，用户不明白标准。why-box 里直接说「发了一条实质性反驳」更清楚
  • follower — 单独一个词没有意义。who-box 里写「你的 follower @xxx」更有价值
  • from engage / from ideas.md / from chat — 内部数据来源路径，用户不需要知道。why-box 负责解释触发原因
  • new contact — 意义模糊。why-box 里写「你们从未互动过」更直接

Slot 3 — Time / Urgency（必有，右对齐）

不是 badge 形式，是独立的位置。示例：

  • 2h ago — 基础时间戳
  • ~6h window — 时效窗口倒计时（peach 色，强提示）
  • weekly — Research 周期
  • acted — 已行动状态
实现规则：
  • 每张 Signal 严格遵循 Slot 1 + Slot 2（可选）+ Slot 3 三槽结构
  • Slot 1 Type Badge：色点 + 全大写小字（`● ENGAGE` / `● CREATE` / `● RESEARCH` / `● NETWORK`）。视觉上是元数据，不是主内容。不再带 · 子类型
  • Slot 2 Trigger Chip：常规 pill 样式，全英文 lowercase，文案为人话短语（参见上节）
  • 不引入 Slot 1 / Slot 2 / Slot 3 之外的第 4 类标签
  • Create 三子类型在 CSS class 上保留区分（trend / idea / activity），但 badge 文案统一为 Create，色点统一为 amber
  • Trigger Chip 数量 ≤ 2，超过则只保留最强的两个
  • Chip 文案必须通过「路人测试」：一个从没用过 Tomo 的人看到这个词，能不能立刻知道它在说什么？不能 → 不上 chip，进 why-box
  • Chip 不能重复标题已经说过的事（如标题已写「@lennysan 引用了你」，就不再上 quoted by @lennysan chip）

Signal 背后的生产线（Brief 内部机制）

Signal 是用户看到的卡片。Signal 之所以能浮出来，是因为 Agent 内部把每一个机会都包成一张 Brief（Create Brief / Engage Brief）在走生产线。下图是 4/20 sync 对齐的内部流水：

Brief 生产线（Agent 内部，不暴露用户 UI）：
Capture（抓原始素材）→ Context（拼上下文）→ Angle（选写作/回复角度）→ Decision（判断是否浮出）→ Output（生成 Signal 卡片）

对用户：只有 Output 阶段的 Signal 卡片可见。 Capture / Context / Angle / Decision 是 Agent 内部状态，不在 Cycle 6 UI 中暴露。

Tracing — 中间产物可追踪

Brief 的每个阶段都结构化落地（不是临时内存状态）。同一次 Capture 的结果可被多张 Signal 共享（例如一次趋势扫描同时驱动一张 Create Signal 和多张 Engage Outbound Signal）。这套 tracing 是 Feedback 机制的前提。

Feedback — Agent 自我优化

Yajun 扩展现有 Review 机制到 Brief 层：每天 AI 咀嚼一次所有 Brief 的中间产物（哪些 Capture 被采用、哪些 Angle 被跳过、哪些 Signal 被用户 acted），产出对下一轮 Capture 方向的优化提示。用户不可见，但决定 Signal 质量的进化速度。

Autopilot / Copilot

删除 Autopilot/Copilot 切换。 在 Signals 模型下，Agent 的输出是 Signal 卡片（信息），不是可执行的动作。所有对外行动（发帖、回复、DM）都需要用户确认。Agent 后台持续扫描不需要用户控制——这不是「Autopilot」，这就是产品的默认工作方式。

Anti-Spam 原则（Product Constraints）

反 AI mass reply 的叙事在 2026 年正在变强。用户在 X 上看到明显 AI 生成的批量回复会感到被冒犯 —— 这种内容不但没有增长效果，反而会损害个人品牌。Cycle 06 Tomo 的立场必须明确：Tomo 不是 mass reply 工具，是 curated opportunity scout。 以下约束必须在实现时硬编码：

数量上限（每用户每天）

Signal 类型每日上限说明
Engage Inbound（自己帖子的回复）≤ 5 条只筛出真正值得回的。互动少的回复不进 Signal
Engage Outbound（外部对话）≤ 3 条宁可给 0 条，也不凑数
Create（所有类型）≤ 2 条发帖机会本来就不应该多
Research（周报）按主题，每主题每周 1 条不是日高频信号
Network≤ 2 条/周关系动态本来就慢
每日总 Signal 上限≤ 10 条超过 10 条，用户就麻木了

Bespoke Reply 原则

每条建议回复必须是针对那条具体对话生成的，不能是模板。 实现上 prompt 必须包含：原帖全文 + 回复上下文 + 用户独特角度（来自 VOICE.md）。如果 agent 生成的回复能换到另一条帖子下也通顺，说明回复质量不合格 —— 应该拒绝输出。

Tag 数量硬约束

  • 单条建议帖子/回复内 tag ≤ 3 个账号
  • 连续 3 条帖子不能出现相同的 tag 组合（避免刷屏）
  • Tag 必须有明确理由（原作者 / 产品官方 / founder / DevRel），不能为了曝光乱 tag

Signal 质量底线（Reject 条件）

以下情况 Agent 应该 主动不生成 Signal，而不是凑数：

  • Engage：原帖下已有 > 100 条回复，且没有独特切入点 → 不推荐用户去挤
  • Engage：找不到用户 voice 里对应的独特视角 → 不生成 Signal 而不是生成平庸回复
  • Create：趋势扫描找到的话题和用户 VOICE.md 只是字面匹配，没有实质角度 → 不生成
  • Create C1/C2/C3：Chat seed 太模糊（如「今天看了一些 AI 新闻」）→ 在 Chat 里反问具体是什么，而不是硬生成

品牌立场（可对外传播）

Tomo 的产品承诺（建议写进 landing page + Settings 页）：
  • ✗ 不代你自动发帖
  • ✗ 不代你自动回复
  • ✗ 不做 mass reply，每条建议都是针对那条对话的
  • ✓ 每天最多 10 个 Signal，宁缺毋滥
  • ✓ 你始终是发送按钮的那个人

通知策略

信号类型Push 通知邮件 Fallback
Engage（时效性）立即推送24h 未打开 → 汇总邮件
Create（趋势窗口）推送不发邮件
Research（周报）不推送可选周报邮件

核心逻辑：通知 = Signal 的推送版。 高时效 → push，低时效 → app 内静默。一天 1-3 条推送，不多。

Onboarding — 冷启动

Day 0 要让 Signals 能用

Cycle 06 产品的所有价值都压在 Signals 上。但 Signals 的质量完全取决于 Agent 对用户的理解是否到位 —— 如果 VOICE.md 是空的，Agent 不知道你的角度，生成的 Signal 就是通用推荐，和不用 Tomo 没区别。

Onboarding 的任务不是定义用户是谁，而是建立一个「足够让 Signals 能工作」的薄初始理解。 Day 0 之后，Agent 通过 Chat seeds、用户 action 反馈、X history 持续校准，理解越来越准。

硬约束（4/20 sync 对齐）：Onboarding 状态下不生成 Signal。
用户只有在 Onboarding 走完 Step 4（VOICE.md 保存）之后，Agent 才启动第一次 Signal scan。Onboarding 进行中：Agent 可以后台扫 X history（Step 2 auto-infer），但不产出任何 Signal 卡片、不写入 Signals feed、不触发通知。避免未对齐 voice 产出的 Signal 污染用户首屏信任。

从 Cycle 05 继承的核心原则（仍然成立）

之前 Cycle 04-05 Onboarding PRD 里定下的几条原则，在 Cycle 06 依然是正确方向，不需要推翻：

1. 先读，再问 — 如果用户 X 上有历史帖子，Agent 必须先 scan + 分析，再问带证据的判断题，不要默认开放长答。

2. 选择优先，表达补充 — 默认 2-3 个选项的判断题，必要时一条轻量自定义。不要默认开放问答。

3. 学习偏好，不急着定义身份 — 学偏好（写什么方向、用什么方式、目标是什么），不要急着归纳「你是谁」。

4. 用户选择高于历史内容 — 历史是证据不是宿命。用户要转方向，系统必须服从。

5. 初始模型要薄，但必须可工作 — Day 0 VOICE.md 可以很薄，但不能空到 Signal 明显跑偏。

6. 深度理解在合作中长出来 — 真正的 voice model 不应该在 Onboarding 里一次建完，应该通过 heartbeat + Chat seeds + action 反馈持续校准。

Onboarding 流程（Cycle 06 版）

Step 1 — Sign Up + Connect X

邮箱注册 → OAuth 连接 X 账号。连接后 Agent 立刻后台开始拉取用户最近 50-100 条 post（如果有）。

  • 预计时间：~1 分钟
  • 产出：OAuth token、user profile、X history 原始数据
  • 阻塞性：硬阻塞（没连 X，Tomo 什么都做不了）

Step 2 — X History Auto-Infer（先读）

Agent 在后台分析用户的 X 历史，产出初步判断，不打扰用户。同时用户看到一个 loading 页面说「正在学习你的风格，约 30 秒」。

自动推断的字段：

  • Topics — 高频出现的 3-5 个话题方向
  • Voice profile — 句式倾向（长/短句）、语气（aggressive/measured/warm）、词汇偏好
  • Anti-patterns — 用户从不用的词、不写的话题
  • Audience — 从互动对象反推
如果用户 X 历史 < 10 条，跳过 auto-infer，直接进 Step 3 用默认问题兜底。

Step 3 — Guided Confirmation（选择式访谈）

不是开放问答，是判断题 + 极少量补充。时间控制在 5-8 分钟。默认 5-6 题。

问题形式产出字段
「我看到你过去更常写这三类内容，接下来先聚焦哪两个？」（附 3 个 auto-infer 的 topic 选项）多选 2/3VOICE.md · Topics
「我读下来你的表达更偏向 A / B，你想继续哪边？」（A=contrarian short-take，B=storyteller + 数据）单选VOICE.md · Voice profile
「你做社媒最想达成的是 A / B / C？」（A=建立影响力，B=找同类人合作，C=给产品/公司带流量）单选VOICE.md · Audience · Goal
「Tomo 应该主要帮你做哪两件？」（Engage inbox / Engage discover / Create trend / Create activity / Research briefing）多选 2/5Signals 类型优先级
「有哪些话题你希望 Agent 持续追踪？」（文本输入，可留空）自由输入Research topics 初始列表
「有什么你绝对不会写的话题或方式？」（可选）自由输入VOICE.md · Anti-patterns

Step 3 的交互形式：Chat Interview Mode

实现方式：直接复用 Chat + Interview Mode，不做额外组件。
Onboarding 就是 Interview Mode 的第一次使用场景。新用户完成 Connect X 后，进入 Chat 页面，Agent 以 Interview Mode 启动 Onboarding 问答。完整交互模式见 Interactive Mock tab → Chat 页 → Interview Mode 段落。

Refresh Voice — 持续校准入口

VOICE.md 不是 Day 0 做完就结束。Cycle 06 Onboarding 只是起点，持续校准通过三个渠道：

1. Chat Seeds 自动累积

用户在 Chat 里说的每一句话都进 memory/。每 30 天 Agent 检查一次 VOICE.md 是否和最近的 Chat seeds 一致 —— 如果用户最近在聊 fintech 但 VOICE.md 只有 AI topic，Agent 会触发一个 Signal 建议更新。

2. 用户 action 反馈

Signal 被采纳 / 跳过 / 编辑的数据是最强的校准信号：

  • 某类 Signal 用户持续跳过 → 该类型降低优先级
  • 用户编辑回复时总往某方向改 → 更新 Voice profile
  • 某类话题的 Create Signal 每次都被采纳 → 加入 Topics

3. Voice Learning Signal（L8）

这是 Cycle 05 通知规则里的 L8。在 Cycle 06 重新定义为：当 Agent 检测到 VOICE.md 和用户实际行为 drift 时，生成一条 Signal（非 email，放在 Signals 页里），让用户一键进入 Refresh Voice 流程。

Engage — 参与对话

从「帮你写」到「帮你回」

Engage 是 Signals 的核心类型，也是第一个上线的功能。在 X 上，最有效的增长方式不是发帖，而是在对的时间回复对的帖子。

Agent 替你扫描，帮你筛选，给你建议回复。你编辑后一键发送。

两个层次

Inbound — 被回复

「有人在你的帖子下说了值得回复的话」

你的 Follower 的回复 · 任何人的回复 · 被引用推文 · 高质量回复 vs 简单点赞的区分

Outbound — 主动发现

「外面有个对话你应该加入」

你关注领域的热门讨论帖 · 有人提到了你 · 趋势中你有独特角度的话题

命名约定（2026-04-20 sync 对齐）： 内部 / 代码 / 文档统一用 Inbound（被回复）和 Outbound（主动发现）。先前的 Layer A / Layer B 已废弃。面向用户的 UI 文案（section header、filter tab）暂沿用 INBOX · WORTH REPLYING 和 DISCOVER · WORTH JOINING，待另行决定是否替换为更口语的表达。

为什么先做 Inbound

Inbound 数据确定性高 — 是你自己的帖子，不存在搜错的问题。
Inbound 用户感知强 — 「有人回了你」比「外面有个对话」更紧迫。
Inbound 验证了核心 UX 循环 — Signal 卡片 → 查看详情 → 采取行动。
Inbound 复用了现有代码 — reply 生成已经做过，只是触发方式变了。
Outbound 新增依赖（4/20 sync）： Outbound Brief 的 Angle 生成需要读回复者的 profile / 近期发帖（和 Inbound 只需读原帖不同）。这意味着 Outbound 比 Inbound 多一层 profile API 调用。实现时通过后端链式 API（Twitter 第三方 → DateX → Official）对 Agent 透明，Agent 层只请求"读 profile"即可。

Engage Brief 数据模型（4/20 sync 定稿）

每一张 Engage Signal 卡片背后对应一个 Engage Brief 对象。和 Create Brief 一样，Cycle 6 Engage Brief 只有核心几字段 —— Inbound 和 Outbound 的区别只在 Reference 的构成 和 生成 Angle 时读哪个 profile。

Variant
Inbound（被回复） vs Outbound（主动发现）。两者共用 Why + Angle + Draft Reply 结构，只在 Reference 字段和 profile 读取要求上不同。
Reference (Inbound)
{ original_tweet, reply_posts[] }。original_tweet = 用户自己的原帖；reply_posts = Tomo 从该帖回复中筛出的高质量回复（1 条或多条，每条带回复者 profile hint）。Inbound 比 Outbound 多一层 API 调用：为每个 reply_post 读取回复者的 profile / 近期发帖，才能为"回 Ta"这个动作生成有针对性的 Angle 和 Draft Reply。
Reference (Outbound)
{ original_tweet, author_context? }。original_tweet = 外部值得加入的一条帖子（趋势中、被你 @ 的、关注账号新帖等）；author_context 只在需要时读取原帖作者的 profile。Outbound 的搜索策略是关键（怎么搜出值得加入的外部帖子），这部分由 Agent 内部 Capture 阶段决定，不暴露到 Brief 字段。
Why
一句大白话说清楚"为什么这条值得回"。Inbound 可能是 "48k 粉丝实质反驳，值得续聊"；Outbound 可能是 "VC 在推测，缺第一手使用数据 —— 你有"。
Angle
建议回复的切入角度，从 VOICE.md + Reference 交叉生成。Cycle 6 先做 1 个 Angle/reply。Inbound 多 reply 情况下每条 reply 各自有独立的 Angle。
Draft Reply
建议的回复正文。和 Create Brief 同规则：异步按需生成。Signal 卡片浮出时 不 预生成 —— 用户在详情面板选中某条 reply（Inbound 多 reply 时）或直接点"让 Tomo 写"（单 reply 时）才实时生成。
关键决策：
  • Inbound 需读回复者 profile。 对每条被筛出的 reply，Agent 要读对方近期发帖/bio 作为 Angle 生成输入 —— 没这层就写不出针对性回复。实现上走后端链式 API（Twitter 第三方 → DateX → Official），对 Agent 透明。
  • Draft Reply 异步生成。 多 reply Inbound 的详情面板里，用户必须先选一条 reply，再点"让 Tomo 写"才触发生成 —— 一上来不为每条 reply 预生成 3 份文本。
  • Bespoke 原则：每条 Draft Reply 必须针对那条具体对话生成。prompt 必须包含：原帖 + reply（Inbound）或原帖 + author context（Outbound）+ VOICE.md 独特角度。可换到另一条帖子下也通顺的回复 = 质量不合格，拒绝输出。

Engage Signal — 卡片合并规则（同一原帖多次 scan）

同一原帖在多次 heartbeat scan 中产出新回复时，不新建卡片，合并更新同一张 Signal 卡片。

设计原则：一张原帖对应一张 Signal。用户的认知单位是「这条帖子需要我回复」，而不是「第一次扫描找到 3 条，第二次又找到 2 条」。Feed 里出现两张同一个帖子的卡片会让用户困惑，且浪费 feed 空间。

Create Signal — Scan 行为（一次性时机窗口）

Create Signal 和 Engage Signal 的 scan 模型根本不同。Engage 的认知单位是「一条持续增长的对话线」，所以 UPSERT 是自然的。Create 的认知单位是「一个时机窗口」——窗口要么开着、要么关了。一旦窗口关闭，rescan 不会让它复活；只有下一次真正的新机会才能生成新卡片。

核心规则：对 Create Signal，scan 只能做两件事 — 新建卡片，或让卡片过期。不会 re-surface（重新上浮）已有卡片。

Create — 内容时机

不是帮你写，是告诉你「现在写什么」

Create 是一个 idea 信号放大器。当前 Tomo agent 已经有完整的 idea pipeline（趋势扫描 → ideas.md → content-craft）。Create Signal 把「时机」维度加进来：这个话题现在正在爆发，你有独特角度，今天发比明天发效果好 10 倍。

Create Brief 数据模型（4/20 sync 定稿）

每一张 Create Signal 卡片背后对应一个 Create Brief 对象。Cycle 6 的 Brief 只有三个核心字段，其他一切（tag、metrics、来源类型）都是附加元数据，不是 Brief 主体。

Why
一句大白话说清楚"为什么值得在当下发这条"。面向用户语言，不是系统指标。
示例：「Claude 5 今早发布定价，你昨天 ship 了一个基于它的 demo — 今天发 before 48h 窗口关闭」
Reference
触发 Brief 的外部引用，带 type 字段标注类型。
字段：{ type: "tweet" | "news" | "trend" | "idea" | "chat_seed", url?, snippet, author? }
UI 在卡片详情中以"原素材块"呈现，一条 Brief 只有 1 条 Reference（多条说明应该拆成多个 Brief）。
Angle
写作角度—— 从用户 VOICE.md + Reference 交叉生成的切入点。
Cycle 6 先做 1 个 Angle（MVP），后续可扩展成多 Angle 候选（让用户选 A/B）。
Angle 字段：{ description, tags?: string[] }，不包含已生成好的推文正文（推文是异步按需生成，见下）。
关键决策：
  • Brief 生命周期在给用户 Signal 后结束。 Agent 不跟踪用户点了 Signal 之后的后续行为（是否发、怎么改、有没有互动）—— 后续行为反馈通过 Stats / Engage metrics 另一条链路，Brief 这头解耦。
  • 建议推文异步生成。 Brief 生成时 不 预生成完整推文正文；Signal 卡片浮出时只展示 Why + Reference + Angle。用户在详情面板点 「让 Tomo 写」 按钮时才实时生成推文正文（~1s），避免预生成浪费算力、避免用户没看就被老内容占位。

Tag 机制 — 算法放大的关键

X 算法的一个关键复利机制：当你和某个大 V 发生实质互动（回复、提及、被引用），算法后续会把你的帖子也推给 TA。所以 Signal 卡片在建议帖子时，必须包含明确的 tag 建议（谁 + 为什么）—— tag 不是礼貌，是放大路径。
场景Tag 对象为什么
学习（C1）原作者（podcast host / article author）作者转发率 ~30-50%，即使不转发也会看到你
反馈（C2）产品官方 + founder早期产品 founder 几乎一定会回复（~90%）
Build（C3）模型/API 官方 + DevRel leadDevRel 主动在找用户案例，官方转发 = 曝光倍数

趋势帖（A1）可选，不强制避免过度 tag 显得功利

Tag 识别的实现

  • 建立 知名账号字典：lab accounts（@AnthropicAI, @OpenAI, @GoogleAI...）+ 已知 DevRel lead + 知名 podcast host
  • Chat seed 解析时做 entity extraction：产品名 → 官方账号；播客名 → 主持人账号
  • 置信度分档处理：
    • High（字典命中 / 官号确认）→ 直接填入 tag
    • Medium（有候选但不唯一）→ 卡片显示 2-3 个候选 @，用户一键选
    • Low（完全找不到）→ 不生成此 Signal。宁缺毋滥。可在 Chat 里反问一次用户确认，存入 memory，下次升为 High
  • Tag 数量限制：单条帖子 ≤ 3 个 tag（超过显得垃圾）

Create 的关键词局限性是 by design

Category A 的关键词来自 VOICE.md topics + Competitor 列表。这是一个封闭集，刻意的。超出认知边界的发现是 Research 的职责，不是 Create 的。

当 Chat seed 太模糊，Agent 应该反问而不是硬生成 Signal：

用户：「今天看了一些 AI 新闻」
Agent：「能具体说说是哪些？是 model releases、funding news、还是别的？这样我可以更精准地生成 Create Signal。」

这是 Cycle 04-05 没有的行为。Agent 以前会硬写一条关于「AI 新闻」的帖子，现在会先问清楚。配合 Anti-Spam 原则。

3. 系统配置变更确认

触发Agent 回复示例
用户在 Settings 修改 VOICE.md topics「Topics 更新：加入 fintech × AI。下次 scan 会覆盖这个方向。」
用户开启/关闭某 Signal 类型「已关闭 Research Signal 推送，Research Briefing 仍会在 Signals 页可见。」
用户调整通知偏好「Push 通知已关。时效性 Signal 会 fallback 到 email。」

为什么在 Chat 不在 toast：这些确认是 Agent 对用户请求的回应，属于对话上下文的一部分。用户以后 scroll 回去能看到「我什么时候改了什么」。

4. 系统错误 / Auth 问题（hard errors）

这些不能用 Signal 卡片表达，因为不是关于 X 内容的信号，是关于系统状态的。

错误类型Chat 消息示例需要用户操作
X token 过期「你的 X token 过期了。我暂时停止 scan。请在 Settings 重新连接。」✓
X API rate limit「X API 限流中，下次 scan 延迟到 X 分钟后。」✗ 仅告知
Signal 生成失败「过去 6h 没有符合质量标准的 Signal。这是正常的，宁缺毋滥。」✗ 仅告知

5. Agent Proactive 询问（低频，可选）

Agent 偶尔基于累积数据主动提问，帮用户做配置决策。频率 ≤ 每周 1 次。

「我注意到你最近 3 周关于 AI agent reliability 的话题互动特别高。要不要我把这个设为 Research 主题开始追踪？」
  • 触发条件必须明确（有数据支撑，不是凭感觉）
  • 永远是「问问题」，不是「通知已做」
  • 用户回复「好的」→ Agent 执行 → 回到类别 3 的确认回执

Chat 不应该出现的消息类型

明确禁止出现在 Chat 里的内容：
  • ❌ Engage/Create/Research/Network Signal 的推送（一律去 Signals 页 + push notification）
  • ❌ 「你该发帖了」这类催促
  • ❌ 「Agent 今天发现了 5 个机会」这类计数通知（Signals 页 badge 已经表达了）
  • ❌ Performance 数字汇报（去 Stats 或 Email）
  • ❌ 任何带「草稿」「task」「publish」字样的消息（这些概念已废弃）
  • ❌ 市场营销内容（如「试试我们的新功能」— 去 app 内 banner 或 email）

  • 活跃用户：每周打开 Chat 2-5 次，每次给 1-2 个 seed
  • Chat 里 Agent 主动消息 ≤ 1 条/周（clarifying question 或 proactive 询问）
  • 错误消息：仅在实际有 auth / rate limit 问题时出现

Cycle 06 — Monetization

Blurred Signals → Unlock → 7-Day Trial

Cycle 06 的付费模型和 Signals 产品模型一体设计。核心思路：Agent 免费做出价值（signals），但价值被"隐约可见"地挡住。用户主动点击 unlock 才开启 7 天免费试用。

核心规则（3 条）

触发点
Onboarding 完成后。Agent 做一次初始 scan，产出首批 signals 进入 Signals feed。Trial 并不自动启动。
可见性
Signals feed 完整渲染（卡片数量、类型 badge、时间戳都可见），但标题 + 内容用 CSS blur 模糊处理。用户隐约看到"有东西在等我"，但看不清。
解锁
Feed 中央或顶部显示 "Unlock — 7-day free trial"。点击调用 POST /api/v1/billing/start-trial。Trial 一次性，用户账号生命周期只能开启一次。

心理学：为什么模糊而不是遮挡

传统 SaaS paywall 是"封锁 → 付费解锁"，给用户的心理感受是"这个东西我还没拥有"。Cycle 06 反过来：让用户先看见"Tomo 已经为我做了 X 件事"，再用模糊制造"我差一步就能看清"的张力。

这是 loss aversion 而不是 cost-benefit 计算。关键条件：

  • 卡片数量必须真实且可信（5–10 条起）
  • 类型 badge、时间戳、发送者头像等外围元数据都不模糊 → 暗示"内容是真的"
  • 模糊强度要刚好看不清内容，但能辨认"这里有字"（6px blur 量级）
  • 解锁 CTA 的文案是"Unlock"，不是"Subscribe"或"Upgrade"

为什么不持续扫描非付费用户

初始 scan 之后 agent 对 non-trialing / non-active 用户完全停止扫描。原因：

  • X API 是最大成本项（$11–76/user/month）。如果非付费用户也持续扫描，10 个非付费用户 ≈ 1 个付费用户的成本 —— 直接吃掉 margin。
  • Cycle 06 的"模糊"付费转化逻辑只需要初始 scan 产出的 signals 存量即可：用户看到的是一个快照，不是流水。
  • 刷新 / 召回靠 email，不靠 agent —— email 成本 < $0.001/封，数量级差距。

API 端点

GET /api/v1/billing/status
JWT。返回 {status, daysLeft, plan}。status = trialing / active / expired。
POST /api/v1/billing/start-trial
JWT。启动 7 天 trial。如果 trial_started_at 已存在 → 返回 409 TRIAL_ALREADY_USED。成功返回新的 BillingStatus。
POST /api/v1/billing/checkout-session
JWT。入参 {priceId}（必须是 monthlyPriceID 或 annualPriceID 之一，否则 400）。返回 Stripe Checkout URL。
POST /api/v1/billing/portal
JWT。返回 Stripe Customer Portal URL，用户在 Stripe 侧管理订阅 / 取消 / 更新支付方式。
POST /api/v1/stripe/webhook
无 JWT，用 Stripe-Signature 头验签。处理 customer.subscription.created/updated/deleted。
GET /api/v1/agent/:account_id/billing/status
Server-to-server（tomo_agent 调用）。返回 status + scan_interval_seconds。非 trialing / non-active 时 agent 完全不扫描。

后端不做 402

Cycle 06 特意去掉了 RequireActiveSubscription middleware 对 /api/v1/bot/* 的硬拦截。原因：

  • Signals API 返回给前端时包含真实内容（前端 blur 只是视觉）。这符合"用户看见价值存在"的心理模型。
  • 如果后端硬拦截，前端就拿不到 signals 的数量、类型、时间戳 —— 那就没东西可模糊了。
  • Trade-off：技术级用户可以 F12 打开 DevTools 看到内容。这是有意接受的。对 99% 的用户无影响，对那 1% 技术用户，他们本来就不是 subscription 目标。
